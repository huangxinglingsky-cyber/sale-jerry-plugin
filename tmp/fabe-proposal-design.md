# FABE 售前方案生成技能 - 设计方案

## 一、核心定位

**Skill 名称**：`fabe-proposal`
**中文名**：FABE 售前方案生成技能
**一句话定位**：将客户的"开放性问题"转化为结构化的 FABE 售前方案，不仅限于产品，也适用于解决方案、方法论、服务能力等维度。

**核心理念**：客户问的往往是模糊的开放问题（如"你们能做什么？""我们现在的监控体系统一不起来怎么办？"），这个 Skill 的价值在于：
1. **拆解**：把开放问题拆解为具体的 F-A-B-E 四层
2. **匹配**：自动关联知识库中的案例、功能、资质作为素材
3. **生成**：输出一份结构完整、可直接使用的售前方案文档

---

## 二、FABE 四层逻辑拆解

| 层级 | 英文 | 核心问题 | 输出内容 | 素材来源 |
|------|------|----------|----------|----------|
| **F - 特征** | Feature | "我们有/能做什么？" | 客观能力描述（产品功能、技术参数、方法论框架、服务内容） | 功能清单 / 知识库 |
| **A - 优势** | Advantage | "为什么比别人好？" | 差异化竞争点（技术领先性、行业深度、实施方法论、服务保障） | 竞品对比 / 经验积累 |
| **B - 利益** | Benefit | "对客户有什么好处？" | 量化价值（效率提升、成本节约、风险降低、合规达标） | 行业数据 / 案例效果 |
| **E - 证据** | Evidence | "怎么证明你说的是真的？" | 第三方验证（客户案例、数据指标、认证资质、POC 结果） | 案例库 / 资质库 |

## 三、参数设计

| 参数 | 类型 | 必须 | 默认值 | 描述 |
|------|------|------|--------|------|
| `customer_question` | string | ✅ | - | 客户的开放性问题或需求描述（≥20字） |
| `customer_name` | string | ✅ | - | 客户企业名称 |
| `proposal_type` | string | ❌ | `auto` | 方案类型：`product`（产品方案）/ `solution`（解决方案）/ `methodology`（方法论）/ `auto`（自动判断） |
| `industry` | string | ❌ | - | 客户所属行业，用于案例匹配和行业术语适配 |
| `context` | string | ❌ | - | 补充上下文（会议记录、之前沟通要点等） |
| `project_name` | string | ❌ | - | 关联项目名称，用于更新项目状态 |
| `depth` | string | ❌ | `standard` | 方案深度：`brief`（精简版）/ `standard`（标准版）/ `comprehensive`（详尽版） |
| `tone` | string | ❌ | `professional` | 输出风格：`professional`（正式商务）/ `conversational`（对话风格）/ `technical`（技术导向） |

**参数示例**：
```json
{
  "customer_question": "我们目前有5个不同的监控工具，数据孤岛严重，运维人员每天要登录好几个系统才能排查一个问题",
  "customer_name": "某城市商业银行",
  "proposal_type": "solution",
  "industry": "金融",
  "context": "客户IT运维部门约50人，已建CMDB但使用率低，上周提到过监控统一的需求",
  "project_name": "某城市商业银行-监控一体化项目",
  "depth": "standard"
}
```

## 四、核心执行步骤（7步）

### 步骤0: 输入校验（防幻觉）
- `customer_question` 是否足够具体（≥20字）
- `customer_name` 是否为空
- 拒绝模糊输入，提示用户补充

### 步骤1: 问题深度解析
- 识别问题类型（技术问题 / 管理问题 / 流程问题 / 综合问题）
- 提取关键词（技术黑话 → 标准术语，可调用 jargon-decoder）
- 推断显性需求 vs 隐性需求
- **输出**：结构化需求分析表

### 步骤2: 方案类型判定
- 若 `proposal_type=auto`，根据问题特征自动判断：
  - 偏技术实现 → product
  - 偏业务场景 → solution
  - 偏方法论/管理 → methodology
- 确定方案框架模板

### 步骤3: 素材检索与关联
- 搜索知识库：
  - 行业案例库（调用 case-matching）
  - 功能清单（若 product 类型，调用 requirement-matching）
  - 企业背景（调用 company-research，若需要）
- 标记素材质量：直接相关 / 行业相关 / 间接参考
- 若关键素材缺失，标注【需补充】并继续

### 步骤4: FABE 结构生成（核心步骤）

#### F层 - 特征描述
- 基于素材，列出 3-5 个核心特征
- 每个特征用一句话概括 + 技术细节补充
- 客观、准确，不夸大

#### A层 - 优势分析
- 对比传统方式 / 竞品方案的差异
- 使用对比表格更直观
- 突出独特性和领先性

#### B层 - 利益转化（关键！）
- 每个 F+A 组合转化为至少1个客户利益
- 利益 = 效率提升 / 成本降低 / 风险减少 / 合规达标
- 尽可能量化（"运维效率提升40%" vs "效率提升"）
- 结合客户具体场景说话（用客户的语言）

#### E层 - 证据支撑
- 匹配的案例（按相关性排序）
- 关键数据指标
- 行业认证 / 资质
- 案例效果数据（来自案例库真实数据）

### 步骤5: 方案文档组装
- 按模板组装完整方案文档
- 方案结构（标准版）：
  1. 背景与需求理解（复述客户问题，体现理解深度）
  2. 方案概述（一句话总结）
  3. FABE 详细方案（核心内容）
  4. 实施路径建议（可选，基于 depth）
  5. 下一步行动建议
- 检查逻辑连贯性

### 步骤6: 质量自检
- ✅ F-A-B 是否形成完整逻辑链（特征→优势→利益）
- ✅ B层是否以客户为中心（而非以产品为中心）
- ✅ E层证据是否真实可溯源
- ✅ 是否引用了知识库中的真实素材
- ✅ 方案语言是否匹配客户行业和沟通风格

### 步骤7: 更新项目状态
- 调用 project-status-updater 记录本次方案生成

## 六、与其他 Skill 的协作关系

```
客户开放问题
    │
    ▼
fabe-proposal（主控）
    ├─→ jargon-decoder    （步骤1：技术黑话解码）
    ├─→ company-research  （步骤1：企业背景调研）
    ├─→ case-matching     （步骤3：案例素材检索）
    ├─→ requirement-matching（步骤3：功能清单匹配，仅 product 类型）
    └─→ project-status-updater（步骤7：更新项目状态）
```

## 七、核心设计亮点

| 亮点 | 说明 |
|------|------|
| **不限于产品** | `proposal_type` 支持 solution / methodology / product 三种类型 |
| **需求深挖** | 步骤1 识别显性需求 + 隐性需求，不是简单复述客户问题 |
| **知识库驱动** | E层证据和F层特征来自真实案例库和功能清单，严禁虚构 |
| **利益导向** | B层强制要求以客户为中心，结合客户场景说话 |
| **灵活深度** | brief / standard / comprehensive 三档，适配不同场景 |
| **质量自检** | 步骤6 内置5项检查，确保输出质量 |

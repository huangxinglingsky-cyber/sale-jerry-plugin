### ⚠️ 需要改进的地方

#### 1️⃣ 临界问题：版本检测逻辑不完整（问题#1）
**位置**：`project-init/SKILL.md` 步骤2.3，第259-266行

**现象**：
- 旧版目录有序号变化（如 `05过程方案` → 新版 `06_方案沟通`）
- 检测代码未考虑这种序号变化
- 可能导致版本识别错误

**后果**：
- 含有序号变化的旧版项目可能被误识别为"未知结构"
- 用户被迫手动处理，而非自动升级

**修复**：
```bash
# 当前（错误的）：
for OLD_DIR in "01项目角色" "02需求文件" ... "09沟通话术"; do
  if test -d "${PROJECT_ROOT}/${OLD_DIR}"; then
    IS_OLD_STRUCTURE=true
  fi
done

# 应该是：
# 使用完整的映射表检测，确保包括所有可能的等效名称
```

---

#### 2️⃣ 临界问题：迁移脚本不够健壮（问题#3）
**位置**：`project-init/SKILL.md` 步骤2.5，第339-358行 中的 migrate_dir() 函数

**现象**：
```bash
rmdir "${OLD_PATH}" 2>/dev/null || echo "⚠️ 旧目录非空..."
```
- 旧目录删除失败时，仅打印警告
- 脚本继续执行，可能导致新旧目录混存

**后果**：
- 迁移不完整：旧目录未被删除
- 混合结构项目：新旧路径同时存在，容易混乱

**修复**：
```bash
# 迁移前验证旧目录内容，迁移后验证删除成功
# 失败时提供明确的恢复步骤
```

---

#### 3️⃣ 主要问题：行业检测实现不清晰（问题#2）
**位置**：`project-init/SKILL.md` 步骤1.4(B)，第191-196行

**现象**：
- 文档说"检查当前工作空间已存在该客户的旧项目"
- 但未说明：
  - 如何提取客户名称
  - 如何查找同客户项目
  - 多个项目时如何选择

**后果**：
- 实施方无法准确理解逻辑
- 可能导致行业检测失败

**修复**：
```bash
# 补充具体实现逻辑：
CUSTOMER_NAME="${INPUT_NAME%%-*}"
find . -maxdepth 1 -type d -name "${CUSTOMER_NAME}-*" ...
```

---

#### 4️⃣ 设计问题：双路径优先级规则不明确（问题#5）
**位置**：所有下游 Skill 中的 fallback 逻辑

**现象**：
- 代码逻辑是"优先新，fallback 旧"
- 但如果项目同时存在新旧目录（迁移不完整），应该如何处理？

**后果**：
- 混合结构项目（迁移中断）可能出现数据混乱
- 文件可能读写到错误目录

**修复**：
```bash
# 添加优先级规则说明：
# 1. 项目含 售前阶段/ → 使用新路径
# 2. 项目不含售前阶段/ → 使用旧路径
# 3. 两者都有 → 警告用户完成迁移
```

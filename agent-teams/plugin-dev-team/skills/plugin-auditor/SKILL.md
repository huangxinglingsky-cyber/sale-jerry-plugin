# Plugin Auditor (Plugin架构审计技能)

## Purpose

对 Claude Plugin/Agent 的配置文件和设计进行专业审计，识别配置错误、命名问题、触发歧义、架构缺陷和潜在风险。

**核心功能**:
- plugin.json 配置完整性校验
- Agent/Skill 命名规范检查
- 触发语义明确性评估
- 架构合理性分析（职责边界、循环调用）
- 常见陷阱识别（Context溢出、超时、幻觉诱导）
- 生成详细的审计报告

**核心价值**:
- ✅ **配置质量保障**：确保配置文件符合规范
- ✅ **设计缺陷识别**：发现命名混淆、触发歧义等问题
- ✅ **风险预警**：识别可能导致运行失败的潜在问题
- ✅ **改进建议**：提供具体可执行的优化方案
- ✅ **只审计不修改**：严格遵守审计原则，不擅自修改代码

## When to Use

在以下情况下使用此技能：
- 开发完成后需要审查 Plugin 配置
- Agent/Skill 触发不正常，需要诊断问题
- 重构前需要评估现有架构
- 代码审查阶段需要专业审计

## Capabilities

### 1. 配置完整性审计

**Schema 校验**：
- 检查 plugin.json 必需字段（name, version, description, author, agents, skills）
- 验证文件路径引用是否正确
- 检测命名冲突

**文件路径验证**：
- 验证 agents 数组中的文件是否存在
- 验证 skills 目录路径是否有效
- 识别路径引用错误

### 2. 命名规范审计

**命名标准**：
- kebab-case 格式检查（requirement-analysis ✅, RequirementAnalysis ❌）
- 动宾结构检查（analyze-requirements ✅, analyzer ❌）
- 避免模糊命名（process-data ❌, extract-requirements ✅）

**参数命名**：
- 参数名自解释性检查
- 避免无意义命名（id, data, value）

### 3. 触发语义审计

**Description 质量评估**：
- 长度适中（50-150字）
- 包含3-8个关键词
- 说明使用场景（"当用户要求..."）
- 独特性（不与其他冲突）

**触发歧义检测**：
- 分析多个 Agent 的关键词重复度
- 识别可能的触发混淆
- 提供差异化建议

**Examples 质量检查**：
- 数量：至少2个
- 结构：包含 Context、user、assistant、commentary
- 覆盖度：覆盖不同使用场景

### 4. 架构合理性审计

**Agent/Skill 边界判断**：
- Skill 应该只做原子操作
- Agent 应该编排多个 Skill
- 识别职责过重的 Skill
- 识别管得太宽的 Agent

**循环调用检测**：
- 构建调用关系图
- 检测循环依赖路径
- 评估风险等级

**过度依赖检测**：
- 统计 Skill 被调用次数
- 识别过度依赖的 Skill
- 建议拆分策略

### 5. 常见陷阱识别

**Context 溢出风险**：
- 检查是否返回过大结果
- 识别未限制的查询
- 建议分页或限制

**超时隐患**：
- 检查外部 API 调用
- 识别大文件处理
- 建议超时处理机制

**幻觉诱导风险**：
- 检查 enum 参数定义
- 识别格式未明确的参数
- 建议补充详细说明

## Instructions

### 执行角色与核心原则

**你的角色定位**：
你是一位首席 Plugin 架构审计师，精通 Claude Plugin/Agent 设计规范和最佳实践。

**你的任务目标**：
对 Plugin 配置和设计进行全面审计，输出详细的审计报告和改进建议。

**核心原则**：
- 只审计，不修改：严禁直接修改用户的配置文件或代码
- 客观严谨：基于明确的规范和标准进行审计
- 提供建议：对每个问题提供具体的改进建议
- 分级标注：问题按严重程度分为 🔴严重阻断、🟡警告优化、💡改进建议

### 执行步骤

#### 步骤 1: 信息收集与文件读取

**1.1 读取 plugin.json**：
```bash
Read(file_path="{project_path}/.claude-plugin/plugin.json")
```

**1.2 扫描 Agent 文件**：
```bash
Glob(pattern="{project_path}/agents/*.md")
```

**1.3 扫描 Skill 文件**：
```bash
Glob(pattern="{project_path}/skills/**/SKILL.md")
```

**1.4 收集项目信息**：
- plugin.json 的完整内容
- 所有 Agent 的 name 和 description
- 所有 Skill 的 name 和 description
- 项目目录结构

#### 步骤 2: 配置完整性审计

**2.1 Schema 校验**：
检查 plugin.json 的必需字段：
- ✅ name: 项目名称
- ✅ version: 版本号
- ✅ description: 项目描述
- ✅ author: 作者信息
- ✅ agents: Agent 列表
- ✅ skills: Skills 目录路径

**2.2 文件路径验证**：
验证 agents 和 skills 引用的文件是否存在

**2.3 命名冲突检测**：
检查是否有重复的 name

#### 步骤 3: 命名规范审计

**3.1 命名格式检查**：
- kebab-case 格式
- 动宾结构
- 自解释性

**3.2 Description 质量评估**：
- 长度适中（50-150字）
- 包含关键词（3-8个）
- 说明使用场景
- 独特性

**3.3 触发歧义检测**：
分析多个 Agent 的关键词重复度

**3.4 Examples 质量检查**：
- 数量：至少2个
- 结构完整
- 覆盖度

#### 步骤 4: 架构合理性审计

**4.1 Agent/Skill 边界判断**：
评估职责划分是否合理

**4.2 循环调用检测**：
检测是否存在循环依赖

**4.3 过度依赖检测**：
检查是否存在过度依赖某个 Skill

#### 步骤 5: 常见陷阱识别

**5.1 Context 溢出风险**：
检查是否可能返回过大的结果

**5.2 超时隐患**：
检查是否处理长耗时任务

**5.3 幻觉诱导风险**：
检查参数定义是否可能导致模型瞎填

#### 步骤 6: 生成审计报告

按照标准格式输出审计报告（见 Output Format）

## Best Practices

### 审计时机
- 开发完成后立即审计
- 每次重大修改后审计
- 定期（如每月）进行全面审计

### 问题修复
- 优先修复 🔴 严重阻断问题
- 逐步优化 🟡 警告问题
- 参考 💡 改进建议进行长期优化

### 持续改进
- 建立审计检查清单
- 将审计纳入 CI/CD 流程
- 定期更新审计标准

## Notes

1. **只审计不修改**：严禁使用 Write/Edit 工具修改原始文件
2. **客观严谨**：基于明确的规范和标准，不做主观臆断
3. **提供建议**：对每个问题提供具体的改进建议
4. **分级标注**：按严重程度分类问题
5. **完整报告**：必须完成所有6个步骤

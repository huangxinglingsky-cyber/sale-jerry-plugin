# Stakeholder Input (项目相关方录入技能)

## Purpose

为项目管理人员提供智能化的项目相关方信息录入能力，自动识别文本或图片中的人员信息，并规范化更新到项目相关方文件中。

**核心功能**:
- 智能文本解析（支持多种格式）
- 图片文字识别（OCR）
- 字段自动提取（姓名、级别、角色、联系方式、邮箱、职责）
- 智能项目识别（自动检测工作空间项目）
- 数据验证和格式化
- 安全文件更新（追加/替换/智能合并）

**核心价值**:
- ✅ **智能识别**：自动提取结构化人员信息
- ✅ **多源支持**：文本、Excel、图片等多种输入
- ✅ **格式规范**：统一的Markdown表格输出
- ✅ **智能项目识别**：自动检测或询问用户选择项目
- ✅ **安全更新**：防止数据丢失的安全更新机制
- ✅ **高效录入**：节省90%的手工录入时间

## When to Use

**自动触发场景**（包含以下关键词时建议使用此技能）：
- 用户输入包含**姓名、电话、手机、联系方式、邮箱**等人员基本信息
- 用户提到**项目成员、团队名单、人员名单、项目组、相关方**等词汇
- 用户上传或提供包含人员信息的**图片、截图、照片**
- 用户说"录入"、"添加"、"更新"项目相关人员信息

**典型使用场景**：
- 收到客户提供的项目组成员名单
- 需要录入会议中获取的人员信息
- 从邮件、微信等渠道收集到的联系人信息
- 项目启动时需要批量录入项目相关方
- 更新现有项目的人员变动信息
- 拍摄的会议白板、纸质名单等需要录入

**关键词识别**：
```
触发关键词（任一即可）：
- 人员信息：姓名、名字、联系人、人员
- 联系方式：电话、手机、座机、联系方式、电话号码
- 电子邮件：邮箱、email、邮件地址
- 组织结构：项目成员、团队、团队名单、项目组、相关方、干系人
- 角色职位：总监、经理、工程师、负责人、对接人、项目经理、CTO
- 操作动作：录入、添加、更新、登记、补充
```

## Capabilities

### 1. 多源智能识别

**支持的输入源**:
- **纯文本**: 结构化文本、列表格式、逗号分隔等
- **Markdown表格**: 直接解析Markdown表格
- **图片**: 通过Read工具读取图片内容（自动OCR识别）
- **混合格式**: 自动识别并提取有效信息

### 2. 字段智能提取

**识别字段**:
- **姓名**: 必填字段，支持中英文姓名
- **级别**: 总监、经理、工程师、专家等职级
- **角色**: CTO、项目经理、技术负责人等具体角色
- **联系方式**: 手机号码（支持多种格式）
- **邮箱**: 邮箱地址（格式验证）
- **职责简要描述**: 工作职责和负责范围

**识别能力**:
- 关键词匹配（姓名、电话、邮箱等）
- 模式识别（手机号、邮箱格式）
- 上下文理解（职责描述提取）
- 缺失字段智能检测和提示补充

### 3. 信息完整性检查

**完整性检测**:
- 自动检测提取信息的完整性
- 识别只有基础信息（姓名+电话）的情况
- 智能提示用户补充关键字段

**补充引导**:
- **触发条件**: 只提取到姓名和联系方式
- **用户选择**:
  - 继续使用当前信息（使用默认值）
  - 补充完整信息（逐项引导输入）
- **引导方式**:
  - 级别选择（总监、经理、工程师等）
  - 角色输入（CTO、项目经理等）
  - 邮箱输入（格式验证）
  - 职责描述输入

**批量处理**:
- 多成员信息不完整时，统一提示一次
- 用户确认后，逐个成员补充信息
- 支持跳过补充，使用默认值

### 4. 数据验证和格式化

**验证规则**:
- 姓名：非空验证
- 手机号：格式验证（支持 138-0000-0000、13800000000 等）
- 邮箱：格式验证（@符号、域名）
- 必填字段检查

**格式化处理**:
- 统一手机号格式（138-0000-0000）
- 邮箱格式标准化
- 职责描述去除冗余字符
- 特殊字符转义（Markdown表格兼容）

### 5. 安全文件更新

**更新模式**:
- **追加模式（append）**: 在现有角色列表后添加新成员
- **替换模式（replace）**: 完全替换现有角色列表
- **智能合并（smart-merge）**: 去重更新（基于姓名+角色）

**安全机制**:
- 更新前备份原文件内容
- 更新时间自动记录
- 保留文件其他部分（项目信息、角色说明等）
- 验证更新结果

## Parameters

| 参数 | 类型 | 必须 | 默认值 | 描述 |
|------|------|------|--------|------|
| input_source | string/object | ✅ | - | 输入源：文本内容或图片文件路径 |
| project_name | string | ❌ | auto | 项目名称（用于定位项目相关方.md文件）；如果不指定，将自动检测工作空间中的项目 |
| update_mode | string | ❌ | append | 更新模式（append/replace/smart-merge） |
| validate_only | boolean | ❌ | false | 仅验证不更新（预览模式） |

**input_source 数据结构**:
```json
// 方式1: 直接提供文本内容
{
  "type": "text",
  "content": "张三 总监 CTO 138-0000-0001 zhangsan@client.com 负责技术决策"
}

// 方式2: 提供图片路径
{
  "type": "image",
  "path": "/path/to/image.png"
}

// 方式3: 简化模式（自动判断）
"张三 总监 CTO 138-0000-0001 zhangsan@client.com 负责技术决策"
```

### 步骤 1: 解析输入源

**执行逻辑**:
```
1. 判断 input_source 类型（文本/图片）
2. 如果是图片：
   - 使用 Read 工具读取图片文件
   - 自动进行 OCR 识别
   - 提取文字内容
3. 如果是文本：
   - 直接使用提供的文本内容
4. 预处理文本（去除特殊字符、统一格式）
```

**输入源类型判断**:
```
如果 input_source 包含以下特征，判定为图片：
- 文件路径（.png, .jpg, .jpeg, .bmp, .gif）
- type: "image"

否则判定为文本内容
```

**图片识别**:
```
使用 Read 工具读取图片文件：
Read(file_path="{图片路径}")

Read 工具会自动进行 OCR 识别并返回文本内容
```

**错误处理**:
- 如果图片读取失败，返回错误信息
- 如果文本内容为空，返回错误信息

---

### 步骤 2: 智能信息提取

**执行逻辑**:
```
从文本中提取项目角色信息：
1. 识别人员条目（按行、按分隔符、按模式）
2. 对每个条目提取字段
3. 验证必填字段
4. 格式化数据
```

**提取模式**:

**模式 1: 表格格式**
```
姓名  级别  角色  联系方式  邮箱  职责
张三  总监  CTO  138-0000-0001  zhangsan@client.com  负责技术决策
```
识别策略：
- 检测表格分隔符（|、\t、多个空格）
- 匹配表头关键词
- 按列提取数据

**模式 2: 列表格式**
```
1. 张三，总监，CTO，138-0000-0001，zhangsan@client.com，负责技术决策
2. 李四，经理，项目经理，138-0000-0002，lisi@client.com，负责项目管理
```
识别策略：
- 检测列表标记（数字、-、•）
- 按逗号或分号分隔字段
- 按顺序映射到字段

**模式 3: 自由文本**
```
张三是我们的CTO，总监级别，电话是138-0000-0001，邮箱zhangsan@client.com，主要负责技术决策和整体技术架构审批
```
识别策略：
- 姓名：句首的人名
- 级别：匹配"总监"、"经理"、"工程师"等关键词
- 角色：匹配"CTO"、"项目经理"等关键词
- 手机号：匹配手机号模式（1\d{10}）
- 邮箱：匹配邮箱模式（\w+@\w+\.\w+）
- 职责：提取"负责"、"主要"后的描述

**字段提取规则**:
```
姓名：
- 优先提取明确标注的姓名
- 匹配中文姓名模式（2-4个汉字）
- 匹配英文姓名模式（首字母大写）

级别：
- 匹配关键词：总监、副总、经理、主管、工程师、专家、顾问
- 未匹配到时使用"未指定"

角色：
- 匹配关键词：CTO、CIO、项目经理、技术负责人、产品经理等
- 提取职位相关词汇

联系方式：
- 匹配手机号模式：1[3-9]\d{9}
- 支持格式：138-0000-0000、13800000000、138 0000 0000
- 格式化为：138-0000-0000

邮箱：
- 匹配邮箱模式：[\w.-]+@[\w.-]+\.\w+
- 验证 @ 符号和域名

职责简要描述：
- 提取"负责"、"主要"、"负责人"后的内容
- 提取"工作内容"、"职责"后的内容
- 限制长度（建议50字以内）
```

**数据验证**:
```
必填字段验证：
- 姓名：必填，不能为空
- 其他字段：可选，未提取到时使用默认值

默认值：
- 级别："未指定"
- 角色："未指定"
- 联系方式："待补充"
- 邮箱："待补充"
- 职责："待补充"
```

```
如果 project_name 参数未提供或为 "auto"：
1. 检测当前工作空间中的项目目录
2. 根据项目数量决定下一步操作：

如果 project_name 参数已提供：
- 直接使用指定的项目名称，跳过检测步骤
```

**项目目录检测**:
```
使用 Bash 工具检测项目目录：

# 方法 1: 查找包含 "01项目角色" 子目录的目录（标准项目结构）
find . -maxdepth 2 -type d -name "01项目角色" 2>/dev/null | sed 's|/01项目角色||' | sed 's|^\./||'

# 方法 2: 查找包含 "项目相关方.md" 文件的项目
find . -maxdepth 3 -name "项目相关方.md" -path "*/01项目角色/*" 2>/dev/null | sed 's|/01项目角色/项目相关方.md||' | sed 's|^\./||'

返回结果示例：
- 农夫山泉-CMDB项目
- 招商银行-监控平台
- 中国银行-DevOps平台
```

### 步骤 4: 更新角色列表

**执行逻辑**:
```
根据 update_mode 参数选择更新策略：

1. append（追加模式）：
   - 保留所有现有成员
   - 在列表末尾追加新成员

2. replace（替换模式）：
   - 清空现有成员列表
   - 使用新成员列表完全替换

3. smart-merge（智能合并）：
   - 基于"姓名+角色"去重
   - 如果存在相同姓名和角色，更新信息
   - 如果不存在，追加新成员
```

**追加模式（append）**:
```
new_list = existing_list + new_members

示例：
现有：张三、李四
新增：王五、赵六
结果：张三、李四、王五、赵六
```

**替换模式（replace）**:
```
new_list = new_members

示例：
现有：张三、李四
新增：王五、赵六
结果：王五、赵六
```

**智能合并（smart-merge）**:
```
for each new_member in new_members:
  found = False
  for each existing_member in existing_list:
    if new_member.name == existing_member.name AND new_member.role == existing_member.role:
      update existing_member with new_member
      found = True
      break
  if not found:
    append new_member to existing_list

示例：
现有：张三(CTO)、李四(项目经理)
新增：张三(CTO, 更新电话)、王五(技术负责人)
结果：张三(CTO, 新电话)、李四(项目经理)、王五(技术负责人)
```

**生成新表格**:
```
生成 Markdown 表格格式：

| 姓名 | 级别 | 角色 | 联系方式 | 邮箱 | 职责简要描述 |
|------|------|------|----------|------|--------------|
| {name} | {level} | {role} | {phone} | {email} | {duty} |
...

⚠️ 特殊字符处理：
- 如果字段包含 |，替换为 \|
- 如果字段包含换行，替换为空格
```

## Best Practices

### 使用最佳实践

1. **推荐使用验证模式**:
   - 首次录入时使用 validate_only=true 预览结果
   - 确认无误后再设置 validate_only=false 执行更新

2. **选择合适的更新模式**:
   - 新增成员：使用 append 模式
   - 全部替换：使用 replace 模式
   - 更新信息：使用 smart-merge 模式

3. **保持输入格式规范**:
   - 使用清晰的分隔符（逗号、制表符、换行）
   - 按照标准顺序排列字段
   - 避免特殊字符和emoji

4. **定期验证数据**:
   - 录入后检查项目相关方.md文件
   - 确认格式正确、数据完整
   - 及时更新变动信息

# Project Init (项目初始化技能)

## Purpose

为B2B销售和实施团队提供标准化的项目目录结构创建能力，确保所有客户项目遵循统一的文档管理规范。

**核心功能**:
- 标准化项目目录创建（9个标准目录）
- 自动生成核心模板文件（项目状态.md、项目相关方.md）
- 项目存在性检查与安全防护
- 多级目录结构自动化生成
- 清晰的执行反馈与错误提示

**核心价值**:
- ✅ **标准化**：所有项目使用统一的目录结构，便于管理和查找
- ✅ **安全性**：防止覆盖已有项目数据
- ✅ **高效性**：一键创建完整的项目结构
- ✅ **规范性**：确保团队遵循统一的文档管理规范
- ✅ **可复用**：统一的目录结构，多个Agent可基于此工作

## When to Use

在以下情况下使用此技能：
- 新客户项目启动时需要创建项目目录
- 需要为客户项目建立标准化的文档管理结构
- 销售团队需要快速准备项目文件夹
- 实施团队需要规范的项目文档组织方式

## Capabilities

### 1. 标准化目录结构

**9个标准目录**（按项目生命周期排序）:

**项目准备阶段**:
- `01项目角色/`: 项目相关方和角色分工信息
- `02需求文件/`: 客户需求文档和需求分析

**销售阶段**:
- `03相关案例/`: 匹配的成功案例
- `04报价文件/`: 商务报价和成本分析

**项目推进阶段**:
- `05过程方案/`: 项目推进过程中的各类文档
- `06会议纪要/`: 客户会议记录和沟通纪要

**实施与签约阶段**:
- `07实施方案/`: 技术实施方案和部署计划
- `08合同文件/`: 正式合同和补充协议

**销售支持**:
- `09沟通话术/`: 销售话术和演示脚本

### 2. 核心模板文件

**自动生成2个核心文件**:

**1. 项目状态.md**（项目根目录）:
- 项目基本信息（名称、创建时间）
- 当前状态标识（立项中/招标中/实施中/已结项）
- 会议纪要记录区
- 状态变更历史表

**2. 项目相关方.md**（01项目角色目录）:
- 项目角色列表表格
- 角色说明（甲方/乙方角色）
- 沟通机制说明

### 3. 安全防护机制

**目录安全检查**:
- ✅ 检查项目根目录是否已存在
- ✅ 使用 `test -d` 命令进行存在性检查
- ❌ 严禁覆盖已存在的项目目录
- ❌ 严禁删除任何已有的项目文件或目录

## Parameters

| 参数 | 类型 | 必须 | 默认值 | 描述 |
|------|------|------|--------|------|
| project_name | string | ✅ | - | 项目名称，支持多种格式：<br>1. "客户名称-项目名称"（如"农夫山泉-CMDB项目"）<br>2. "项目名称"（如"CMDB项目"，自动补充客户名称）<br>3. 自动从环境变量 `JAVIS_WORKSPACE_NAME` 获取客户名称 |
| base_path | string | ❌ | ./ | 项目根目录的基础路径，默认为当前工作目录 |
| skip_templates | boolean | ❌ | false | 是否跳过模板文件生成（仅创建目录结构） |

## Instructions

### 执行角色与核心原则

**你的角色定位**：
你是一位专业的项目初始化助手，专门为B2B销售和实施团队创建标准化的项目文档管理结构。

**你的任务目标**：
快速创建规范的项目目录结构，生成核心模板文件，确保项目文档管理的标准化和安全性。

**核心原则**：
- 安全第一：永远不要覆盖已存在的项目目录

- 清晰反馈：无论成功或失败，都要给用户明确的反馈

### 执行步骤

#### 步骤 1: 参数验证与智能处理

**1.1 检查项目名称参数**
- 检查用户是否提供了 `project_name` 参数
- 如果未提供，返回错误提示

**1.2 智能处理项目名称格式**

使用以下逻辑智能处理项目名称：

```bash
# 获取环境变量 JAVIS_WORKSPACE_NAME（客户名称）
CUSTOMER_NAME="${JAVIS_WORKSPACE_NAME}"

# 获取用户输入的项目名称
INPUT_NAME="{project_name}"

# 智能处理逻辑
if [[ "$INPUT_NAME" == *"-"* ]]; then
  # 情况1: 用户输入包含"-"，如"农夫山泉-CMDB项目"
  # 提取"-"后面的部分作为项目名称
  PROJECT_SUFFIX="${INPUT_NAME##*-}"
  # 最终项目名称 = 环境变量客户名称 + "-" + 提取的项目名称
  FINAL_PROJECT_NAME="${CUSTOMER_NAME}-${PROJECT_SUFFIX}"
else
  # 情况2: 用户输入不包含"-"，如"CMDB项目"
  # 最终项目名称 = 环境变量客户名称 + "-" + 用户输入
  FINAL_PROJECT_NAME="${CUSTOMER_NAME}-${INPUT_NAME}"
fi

echo "处理后的项目名称: ${FINAL_PROJECT_NAME}"
```

**处理示例**：
- 用户输入: "农夫山泉-CMDB项目" + 环境变量: "优维科技" → 最终: "优维科技-CMDB项目"
- 用户输入: "CMDB项目" + 环境变量: "优维科技" → 最终: "优维科技-CMDB项目"
- 用户输入: "华为-DevOps-监控平台" + 环境变量: "优维科技" → 最终: "优维科技-监控平台"

**1.3 环境变量检查**
- 如果环境变量 `JAVIS_WORKSPACE_NAME` 未设置或为空：
  - 如果用户输入包含"-"，直接使用用户输入的完整名称
  - 如果用户输入不包含"-"，提示用户需要设置环境变量或使用完整格式

#### 步骤 2: 安全检查

使用 Bash 命令检查项目目录是否存在:
```bash
test -d "{base_path}/${FINAL_PROJECT_NAME}" && echo "EXISTS" || echo "NOT_EXISTS"
```

**判断逻辑**:
- 如果返回 "EXISTS":
  - ❌ 立即停止执行
  - 📢 向用户输出提示信息:
    ```
    ⚠️ 项目目录已存在: ${FINAL_PROJECT_NAME}

    该项目已经初始化过，为了保护已有数据，初始化过程已停止。

    如需重新初始化，请先手动删除或重命名现有目录。
    ```
  - 🛑 不执行任何后续步骤

- 如果返回 "NOT_EXISTS":
  - ✅ 继续执行

## Best Practices

### 项目命名建议
- ✅ **灵活格式支持**:
  - 完整格式: `客户名称-项目名称`（如"农夫山泉-CMDB项目"）
  - 简化格式: `项目名称`（如"CMDB项目"，自动补充客户名称）
- ✅ **推荐示例**:
  - 用户输入"CMDB项目" → 自动处理为"优维科技-CMDB项目"
  - 用户输入"农夫山泉-CMDB项目" → 自动处理为"优维科技-CMDB项目"
  - 用户输入"监控平台建设" → 自动处理为"优维科技-监控平台建设"
- ✅ **环境变量配置**: 建议在系统中设置 `JAVIS_WORKSPACE_NAME` 环境变量为当前客户名称
- ❌ **避免使用特殊字符**: `/`, `\`, `:`, `*`, `?`, `"`, `<`, `>`, `|`（`-` 除外）
- ❌ **避免过长名称**: 建议长度不超过50个字符

### 环境变量配置
```bash
# 在 ~/.bashrc 或 ~/.zshrc 中添加
export JAVIS_WORKSPACE_NAME="优维科技"

# 或在当前会话中临时设置
export JAVIS_WORKSPACE_NAME="优维科技"
```

### 目录使用建议
1. **保持目录简洁**: 每个目录只存放对应类型的文件
2. **使用子目录**: 如果文件较多，可在二级目录下创建子目录
3. **文件命名规范**: 使用日期前缀，例如: `20250119-需求讨论会.md`
4. **定期归档**: 项目结束后，将整个项目目录归档保存

### 项目状态文件维护
1. **及时更新**: 每次重要会议或状态变化后，立即更新"项目状态.md"
2. **简要记录**: 只记录关键结论，详细内容保存到对应目录
3. **状态同步**: 当项目进入新阶段时，同步更新状态标识
4. **定期回顾**: 定期查看项目状态文件，确保项目进展清晰可追溯

## Notes

1. **安全第一**: 永远不要覆盖已存在的项目目录
2. **清晰反馈**: 无论成功或失败，都要给用户明确的反馈

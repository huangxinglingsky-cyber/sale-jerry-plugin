# Project Init (项目初始化技能)

## Purpose

为B2B销售和实施团队提供标准化的项目目录结构创建能力，确保所有客户项目遵循统一的文档管理规范。

**核心功能**:
- 标准化项目目录创建（售前阶段下11个标准目录）
- 自动生成核心模板文件（项目状态.md、项目相关方.md）
- 项目存在性检查与安全防护
- 旧版目录兼容检测（避免重复创建）
- 多级目录结构自动化生成
- 清晰的执行反馈与错误提示

**核心价值**:
- ✅ **标准化**：所有项目使用统一的目录结构，便于管理和查找
- ✅ **安全性**：防止覆盖已有项目数据
- ✅ **高效性**：一键创建完整的项目结构
- ✅ **规范性**：确保团队遵循统一的文档管理规范
- ✅ **可复用**：统一的目录结构，多个Agent可基于此工作

## When to Use

在以下情况下使用此技能：
- 新客户项目启动时需要创建项目目录
- 需要为客户项目建立标准化的文档管理结构
- 销售团队需要快速准备项目文件夹
- 实施团队需要规范的项目文档组织方式

## Capabilities

### 1. 标准化目录结构

**目录层级**: 项目根目录 → `售前阶段/` → 11个标准子目录

**11个标准目录**（位于 `售前阶段/` 下，按售前生命周期排序）:

**项目准备**:
- `01_项目角色/`: 项目相关方和角色分工信息
- `02_需求文件/`: 客户需求文档和需求分析

**商务拓展**:
- `03_商务案例/`: 匹配的成功案例
- `04_报价文件/`: 商务报价和成本分析

**项目推进**:
- `05_会议纪要/`: 客户会议记录和沟通纪要
- `06_方案沟通/`: 项目推进过程中的各类方案文档

**实施与投标**:
- `07_POC实施/`: POC技术实施方案和部署计划
- `08_标前引导/`: 标前引导策略和执行文档
- `09_商务投标/`: 投标文件和商务标书

**签约与支持**:
- `10_合同文件/`: 正式合同和补充协议
- `11_通用资料/`: 通用销售资料和演示脚本

### 2. 核心模板文件

**自动生成2个核心文件**:

**1. 项目状态.md**（项目根目录）:
- 项目基本信息（名称、创建时间）
- 当前状态标识（立项中/招标中/实施中/已结项）
- 会议纪要记录区
- 状态变更历史表

**2. 项目相关方.md**（售前阶段/01_项目角色目录）:
- 项目角色列表表格
- 角色说明（甲方/乙方角色）
- 沟通机制说明

### 3. 安全防护机制

**目录安全检查**:
- ✅ 检查项目根目录是否已存在
- ✅ 使用 `test -d` 命令进行存在性检查
- ❌ 严禁覆盖已存在的项目目录
- ❌ 严禁删除任何已有的项目文件或目录

### 4. 旧版目录兼容检测

创建各子目录前，先检测项目下是否已存在**名称等效**的旧版目录，若存在则跳过创建（不重复建立）。

**等效判断规则**：忽略序号和下划线，仅比较中文名称部分是否一致。例如：
- `01项目角色` ≡ `01_项目角色`（名称相同）
- `06会议纪要` ≡ `05_会议纪要`（名称相同，序号不同视为等效）

## Parameters

| 参数 | 类型 | 必须 | 默认值 | 描述 |
|------|------|------|--------|------|

### 执行步骤

#### 步骤 1: 参数验证与智能处理

**1.1 检查项目名称参数**
- 检查用户是否提供了 `project_name` 参数
- 如果未提供，返回错误提示

**1.2 智能处理项目名称格式**

使用以下逻辑智能处理项目名称：

```bash
# 获取环境变量 JAVIS_WORKSPACE_NAME（客户名称）
CUSTOMER_NAME="${JAVIS_WORKSPACE_NAME}"

# 获取用户输入的项目名称
INPUT_NAME="{project_name}"

# 智能处理逻辑
if [[ "$INPUT_NAME" == *"-"* ]]; then
  # 情况1: 用户输入包含"-"，如"农夫山泉-CMDB项目"
  # 提取"-"后面的部分作为项目名称
  PROJECT_SUFFIX="${INPUT_NAME##*-}"
  # 最终项目名称 = 环境变量客户名称 + "-" + 提取的项目名称
  FINAL_PROJECT_NAME="${CUSTOMER_NAME}-${PROJECT_SUFFIX}"
else
  # 情况2: 用户输入不包含"-"，如"CMDB项目"
  # 最终项目名称 = 环境变量客户名称 + "-" + 用户输入
  FINAL_PROJECT_NAME="${CUSTOMER_NAME}-${INPUT_NAME}"
fi

echo "处理后的项目名称: ${FINAL_PROJECT_NAME}"
```

**⚠️ 中间输出一致性校验（强制执行）**:

> 在名称处理过程中，确保所有中间输出和最终输出使用**同一个变量 `FINAL_PROJECT_NAME`**。禁止在思考阶段和最终执行之间出现名称不一致的情况。

**一致性规则**:
```
✅ 步骤 1 确定的 FINAL_PROJECT_NAME 是唯一的项目名称
✅ 步骤 2-7 中所有引用项目名称的地方必须使用 FINAL_PROJECT_NAME
✅ 面向用户的输出（步骤 7）中显示的项目名称必须与 FINAL_PROJECT_NAME 完全一致
🚫 禁止在输出中出现与 FINAL_PROJECT_NAME 不同的项目名称
🚫 禁止在步骤 7 的"项目名称处理说明"中显示未经处理的原始名称作为最终名称
```

**输出一致性校验清单**:
```
□ 步骤 7 输出中的"项目初始化完成: {名称}" 是否与 FINAL_PROJECT_NAME 一致？
□ 步骤 7 输出中的"项目路径" 是否与 FINAL_PROJECT_NAME 一致？
□ 步骤 7 的"项目名称处理说明"中"最终项目名称" 是否与 FINAL_PROJECT_NAME 一致？
□ 项目状态.md 中写入的项目名称是否与 FINAL_PROJECT_NAME 一致？
```

**处理示例**：
- 用户输入: "农夫山泉-CMDB项目" + 环境变量: "优维科技" → 最终: "优维科技-CMDB项目"
- 用户输入: "CMDB项目" + 环境变量: "优维科技" → 最终: "优维科技-CMDB项目"
- 用户输入: "华为-DevOps-监控平台" + 环境变量: "优维科技" → 最终: "优维科技-监控平台"

**1.3 环境变量检查**
- 如果环境变量 `JAVIS_WORKSPACE_NAME` 未设置或为空：
  - 如果用户输入包含"-"，直接使用用户输入的完整名称
  - 如果用户输入不包含"-"，提示用户需要设置环境变量或使用完整格式

**1.3.1 环境变量为工作空间标识时的处理规则**⚠️

> 当 Agent 判断 `JAVIS_WORKSPACE_NAME` 的值为工作空间标识（如 "my-workspace"、"workspace-001"、"测试环境"等）而非客户名称时，**必须按以下规则处理**，不得自作主张改变名称提取逻辑。

**判断标准**：
```
以下情况视为"工作空间标识"而非"客户名称"：
- 包含 "workspace"、"工作空间" 关键词
- 包含数字编号且无企业特征（如 "ws-001"、"space-3"）
- 包含 "test"、"dev"、"prod" 等环境标识
- 为通用词汇（如 "demo"、"default"）
```

**处理规则**：
```
IF JAVIS_WORKSPACE_NAME 被判断为工作空间标识（非客户名称）:
    # 策略：忽略环境变量，退回到用户输入本身
    IF 用户输入包含 "-":
        FINAL_PROJECT_NAME = 用户输入的完整名称  # 直接使用，不替换前缀
    ELSE:
        # 环境变量不可用，用户输入也无"-"
        → 返回错误，要求用户提供完整格式（"客户名称-项目名称"）
        → 不得将工作空间标识作为客户名称拼接
ELSE:
    # 正常流程：环境变量为客户名称
    → 按步骤 1.2 的标准逻辑处理
```

**⚠️ 强制规则**:
```
🚫 禁止将工作空间标识（如 "my-workspace"）作为客户名称拼接到项目名称中
🚫 禁止在判断为工作空间标识后仍然执行标准名称提取逻辑
🚫 禁止向用户输出环境变量的具体值（即使判断为工作空间标识，也不泄露变量内容）
✅ 环境变量仅在内部判断使用，其值不得出现在任何面向用户的输出中
```

#### 步骤 2: 安全检查与版本检测

**2.1 检查项目目录是否存在**

```bash
test -d "{base_path}/${FINAL_PROJECT_NAME}" && echo "EXISTS" || echo "NOT_EXISTS"
```

**2.2 判断逻辑**:

- 如果返回 `NOT_EXISTS`:
  - ✅ 继续执行步骤 3（创建全新目录结构）

- 如果返回 `EXISTS`:
  - 进入步骤 2.3，检测是旧版结构还是新版结构，**不要立即停止**

---

**2.3 检测目录结构版本**

当目录已存在时，使用以下逻辑判断版本：

```bash
PROJECT_ROOT="{base_path}/${FINAL_PROJECT_NAME}"

# 判断是否为新版结构（存在 售前阶段/ 子目录）
IS_NEW_STRUCTURE=false
if test -d "${PROJECT_ROOT}/售前阶段"; then
  IS_NEW_STRUCTURE=true
fi

# 判断是否为旧版结构（根目录下存在已知旧版或过渡版子目录，但无 售前阶段/）
IS_OLD_STRUCTURE=false
# 检查经典旧版目录名（无下划线、无序号变化）
for OLD_DIR in "01项目角色" "02需求文件" "03相关案例" "04报价文件" \
               "05过程方案" "06会议纪要" "07实施方案" "08合同文件" "09沟通话术"; do
  if test -d "${PROJECT_ROOT}/${OLD_DIR}"; then
    IS_OLD_STRUCTURE=true
    break
  fi
done
# 同时检查"过渡状态"：目录已按新式命名（含下划线）但仍在根目录（未迁移进 售前阶段/）
if ! $IS_OLD_STRUCTURE; then
  for TRANS_DIR in "01_项目角色" "02_需求文件" "03_商务案例" "04_报价文件" \
                   "05_会议纪要" "06_方案沟通" "07_POC实施" "10_合同文件" "11_通用资料"; do
    if test -d "${PROJECT_ROOT}/${TRANS_DIR}"; then
      IS_OLD_STRUCTURE=true
      break
    fi
  done
fi
```

**版本判断结果处理**:

| IS_NEW_STRUCTURE | IS_OLD_STRUCTURE | 判定 | 处理方式 |

**迁移脚本**：

```bash
PROJECT_ROOT="{base_path}/${FINAL_PROJECT_NAME}"
PHASE_DIR="${PROJECT_ROOT}/售前阶段"

# 1. 创建 售前阶段 中间目录
mkdir -p "${PHASE_DIR}"

# 2. 逐一迁移旧目录
migrate_dir() {
  local OLD_NAME="$1"
  local NEW_NAME="$2"
  local OLD_PATH="${PROJECT_ROOT}/${OLD_NAME}"
  local NEW_PATH="${PHASE_DIR}/${NEW_NAME}"

  if ! test -d "${OLD_PATH}"; then
    return 0  # 旧目录不存在，跳过
  fi

  if ! test -d "${NEW_PATH}"; then
    # 首选路径：新目录不存在时直接重命名（原子操作，自动包含所有内容含隐藏文件和子目录）
    if mv "${OLD_PATH}" "${NEW_PATH}"; then
      echo "✅ 迁移: ${OLD_NAME}/ → 售前阶段/${NEW_NAME}/"
    else
      echo "❌ 迁移失败: ${OLD_NAME}/ — 请手动将该目录移动到 售前阶段/${NEW_NAME}/"
    fi
  else
    # 兜底路径：新目录已存在（上次迁移中断的残留），逐项移动内容
    # 使用 find 遍历（含隐藏文件），避免 glob 展开问题
    find "${OLD_PATH}" -maxdepth 1 -mindepth 1 -exec mv {} "${NEW_PATH}/" \; 2>/dev/null
    if rmdir "${OLD_PATH}" 2>/dev/null; then
      echo "✅ 迁移: ${OLD_NAME}/ → 售前阶段/${NEW_NAME}/"
    else
      echo "❌ 旧目录 ${OLD_NAME}/ 仍有残留内容（可能含子目录），请手动将剩余内容移到 售前阶段/${NEW_NAME}/ 后删除旧目录"
    fi
  fi
}

migrate_dir "01项目角色"  "01_项目角色"
migrate_dir "02需求文件"  "02_需求文件"
migrate_dir "03相关案例"  "03_商务案例"
migrate_dir "04报价文件"  "04_报价文件"
migrate_dir "06会议纪要"  "05_会议纪要"
migrate_dir "05过程方案"  "06_方案沟通"
migrate_dir "07实施方案"  "07_POC实施"
migrate_dir "08合同文件"  "10_合同文件"
migrate_dir "09沟通话术"  "11_通用资料"

# 3. 补建迁移后仍缺失的新增目录
for NEW_DIR in "08_标前引导" "09_商务投标"; do
  mkdir -p "${PHASE_DIR}/${NEW_DIR}"
  echo "✅ 新建: 售前阶段/${NEW_DIR}/"
done
```

迁移完成后，**跳转到步骤 6**（验证）和步骤 7（输出结果），不再重复执行步骤 3-5。

| meeting-analysis | 文件保存 | 将会议分析保存到已创建的项目目录 |

## Best Practices

### 项目命名建议
- ✅ **灵活格式支持**:
  - 完整格式: `客户名称-项目名称`（如"农夫山泉-CMDB项目"）
  - 简化格式: `项目名称`（如"CMDB项目"，自动补充客户名称）
- ✅ **推荐示例**:
  - 用户输入"CMDB项目" → 自动处理为"优维科技-CMDB项目"
  - 用户输入"农夫山泉-CMDB项目" → 自动处理为"优维科技-CMDB项目"
  - 用户输入"监控平台建设" → 自动处理为"优维科技-监控平台建设"
- ✅ **环境变量配置**: 建议在系统中设置 `JAVIS_WORKSPACE_NAME` 环境变量为当前客户名称
- ❌ **避免使用特殊字符**: `/`, `\`, `:`, `*`, `?`, `"`, `<`, `>`, `|`（`-` 除外）
- ❌ **避免过长名称**: 建议长度不超过50个字符

### 环境变量配置
```bash
# 在 ~/.bashrc 或 ~/.zshrc 中添加
export JAVIS_WORKSPACE_NAME="优维科技"

# 或在当前会话中临时设置
export JAVIS_WORKSPACE_NAME="优维科技"
```

### 目录使用建议
1. **保持目录简洁**: 每个目录只存放对应类型的文件
2. **使用子目录**: 如果文件较多，可在二级目录下创建子目录
3. **文件命名规范**: 使用日期前缀，例如: `20250119-需求讨论会.md`
4. **定期归档**: 项目结束后，将整个项目目录归档保存

### 项目状态文件维护
1. **及时更新**: 每次重要会议或状态变化后，立即更新"项目状态.md"
2. **简要记录**: 只记录关键结论，详细内容保存到对应目录
3. **状态同步**: 当项目进入新阶段时，同步更新状态标识
4. **定期回顾**: 定期查看项目状态文件，确保项目进展清晰可追溯

## Notes

1. **安全第一**: 对已存在目录不直接覆盖，迁移前必须经用户确认
2. **版本感知**: 区分新版/旧版/未知结构，给出对应处理策略
3. **清晰反馈**: 无论成功、迁移或失败，都要给用户明确的反馈
4. **标准化**: 所有项目使用统一的目录结构，便于管理和查找
5. **扩展性**: 如需增加新的目录类型，可以修改目录列表
6. **旧版兼容**: 创建子目录前先检测等效旧版目录是否已存在，避免重复创建
7. **文件生成策略**:
   - ✅ 自动生成: 仅2个核心模板文件（项目状态.md、项目相关方.md）
   - 📁 空目录: 其余11个目录仅创建文件夹，不生成任何模板文件
   - 📝 按需生成: 其他文件由用户手动创建或通过其他 Skills 生成

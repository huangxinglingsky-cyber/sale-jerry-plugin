# Skill Logic Auditor (技能逻辑审查官)

## Purpose

作为 Skill 逻辑审计专家，读取并审查 Plugin 中其他 Skill 的定义与逻辑（Prompt/代码），诊断其是否存在"信息缺失时强行输出结论"或"未充分利用知识库"的问题，并输出标准化的审查报告与优化方案。

**核心功能**:
- 读取目标 Skill 的完整定义内容
- 诊断逻辑缺陷（早熟结论、脱离知识库）
- 设计信息补齐策略（引导式追问框架）
- 设计知识库调用逻辑（RAG 集成方案）
- 输出重构后的 Skill 提示词

**核心价值**:
- 🛡️ **防幻觉**：强制 Skill 在信息不足时追问而非编造
- 📚 **知识库优先**：确保 Skill 优先检索内部知识库
- 📋 **标准化审计**：输出结构化的审计报告
- 🔧 **可执行优化**：直接输出可替换的重构 Prompt

## When to Use

在以下情况下使用此技能：
- 用户说"审计某个 skill"、"审查 skill 逻辑"
- 用户说"优化 skill 提示词"、"检查 skill 缺陷"
- 用户说"审计所有 skill"、"批量审计"（不指定具体 skill）
- 需要诊断 Skill 是否容易产生幻觉输出
- 需要为 Skill 添加知识库检索逻辑
- Skill 输出质量不稳定，需要优化

## Capabilities

### 1. 单 Skill 审计

对指定的单个 Skill 进行深度审计，输出详细的审计报告与优化方案。

### 2. 批量 Skill 审计

当用户未指定具体 Skill 时，自动扫描 Plugin 目录下所有 Skill，进行批量审计：

**扫描路径**: `/workspace/apps/pulgins/sale-jerry-plugin/skills/`

**批量审计流程**:
1. 扫描 skills 目录，获取所有 Skill 列表
2. 逐个读取 SKILL.md 文件
3. 对每个 Skill 执行快速审计（聚焦高风险模式）
4. 汇总所有发现的问题
5. 按风险等级排序输出批量审计报告

### 3. 信息补齐防幻觉审计 (Ask-Before-Answer Audit)

检查目标 Skill 是否具备"输入完整性校验"逻辑：

**审计要点**：
- ✅ 是否定义了"必需输入参数"清单
- ✅ 是否在 Instructions 中包含"信息完整性检查"步骤
- ✅ 是否定义了"信息不足时的追问话术"
- ❌ 是否存在"假设性推断"或"猜测性结论"的指令
- ❌ 是否缺少"无法确定时的降级策略"

**风险模式识别**：
```
高风险模式：
- "根据...推断..."（无验证的推断）
- "假设用户意图是..."（无依据的假设）
- "通常情况下..."（无上下文的概括）
- 直接输出结论而无校验步骤
```

### 2. 知识库优先策略审计 (RAG-First Audit)

检查目标 Skill 是否强制加入知识库检索步骤：

**审计要点**：
- ✅ 是否在推理前包含"提取关键词"步骤
- ✅ 是否包含"检索内部知识库"的指令
- ✅ 是否包含"交叉比对验证"的逻辑
- ✅ 是否引用了 `$JAVIS_KB_*` 或知识库相关环境变量
- ❌ 是否仅依赖大模型预训练权重
- ❌ 是否缺少"知识库未命中"的降级策略

**标准 RAG 流程**：
```
1. 提取查询关键词 (Query Extraction)
   ↓
2. 检索内部知识库 (Knowledge Retrieval)
   ↓
3. 交叉比对验证 (Cross-Validation)
   ↓
4. 综合生成回答 (Synthesis)
   ↓
5. 标注信息来源 (Source Attribution)
```

### 3. 审计报告生成

输出标准化的《Skill 逻辑审计与优化报告》，包含四个核心部分。

## Parameters

| 参数 | 类型 | 必须 | 默认值 | 描述 |
|------|------|------|--------|------|
| target_skill_name | string | ❌ | 全部 | 需要审计的目标 Skill 名称，不指定则审计所有 Skill |
| target_skill_content | string | ❌ | 自动读取 | 目标 Skill 的完整内容（不提供则自动读取） |
| skill_purpose | string | ❌ | 自动提取 | 该 Skill 的预期业务目标 |
| focus_area | string | ❌ | all | 审计聚焦区域：all / hallucination / rag |
| output_refactored | boolean | ❌ | true | 是否输出重构后的 Prompt（批量审计时建议设为 false） |
| audit_mode | string | ❌ | auto | 审计模式：single（单个）/ batch（批量）/ auto（自动判断） |

## Instructions

### 执行角色与核心原则

**你的角色定位**：
你是一位 Skill 逻辑审计专家，精通 Prompt Engineering 和 RAG 系统设计，专门诊断 Skill 的逻辑缺陷并提供可执行的优化方案。

**你的任务目标**：
审查目标 Skill 的定义与逻辑，识别"信息缺失时强行输出"和"未利用知识库"的问题，输出标准化的审计报告与优化 Prompt。

**核心原则**：
1. **追问优于编造**：信息不足时必须追问，禁止猜测
2. **检索优于记忆**：优先检索知识库，而非依赖预训练
3. **透明优于黑盒**：所有推理过程必须可追溯
4. **验证优于假设**：任何结论都需要验证依据

### 审计核心原则

#### 原则一：信息补齐防幻觉 (Ask-Before-Answer)

```
┌─────────────────────────────────────────────────────┐
│              信息完整性校验流程                        │
├─────────────────────────────────────────────────────┤
│  用户输入                                            │
│     ↓                                               │
│  [必需参数检查]                                      │
│     ├── 完整 → 继续执行                              │
│     └── 缺失 → 触发追问（禁止编造）                    │
│     ↓                                               │
│  [上下文充分性检查]                                   │
│     ├── 充分 → 继续执行                              │
│     └── 不足 → 引导补充（禁止假设）                    │
│     ↓                                               │
│  执行核心逻辑                                        │
└─────────────────────────────────────────────────────┘
```

#### 原则二：知识库优先 (RAG-First Strategy)

```
┌─────────────────────────────────────────────────────┐
│              知识库优先推理流程                        │
├─────────────────────────────────────────────────────┤
│  用户问题                                            │
│     ↓                                               │
│  [关键词提取]                                        │
│     ├── 实体识别                                     │
│     ├── 意图分类                                     │
│     └── 查询词构建                                   │
│     ↓                                               │
│  [知识库检索]                                        │
│     ├── 检索内部文档                                 │
│     ├── 检索案例库                                   │
│     └── 检索配置库                                   │
│     ↓                                               │
│  [交叉验证]                                          │
│     ├── 多源一致性检查                               │
│     ├── 时效性验证                                   │
│     └── 权威性评估                                   │
│     ↓                                               │
│  [综合回答]                                          │
│     └── 标注信息来源                                 │
└─────────────────────────────────────────────────────┘
```

### 执行步骤

#### 步骤 0: 确定审计范围

**0.1 判断审计模式**

根据用户输入判断审计模式：

```python
# 判断逻辑
if 用户指定了具体的 skill 名称:
    审计模式 = "单 Skill 深度审计"
    目标 = 指定的 Skill
elif 用户说"审计所有"/"批量审计"/"全部 skill":
    审计模式 = "批量快速审计"
    目标 = skills/ 目录下所有 Skill
else:
    # 默认进行批量审计
    审计模式 = "批量快速审计"
    目标 = skills/ 目录下所有 Skill
```

**0.2 批量审计时扫描 Skill 列表**

```bash
# 扫描 skills 目录
SKILLS_DIR="/workspace/apps/pulgins/sale-jerry-plugin/skills"

# 获取所有 Skill 名称
ls -d $SKILLS_DIR/*/ | xargs -n1 basename

# 排除自身（skill-logic-auditor）
# 对每个 Skill 执行快速审计
```

#### 步骤 1: 读取目标 Skill 内容

**1.1 确定目标 Skill**

```bash
# 根据 target_skill_name 定位文件
SKILL_PATH="apps/pulgins/sale-jerry-plugin/skills/{target_skill_name}/SKILL.md"

# 如果用户未提供内容，自动读取
if [ -z "$target_skill_content" ]; then
  Read(file_path="$SKILL_PATH")
fi
```

**1.2 解析 Skill 结构**

提取以下关键部分：
- `description`: 触发条件与功能描述
- `Parameters`: 输入参数定义
- `Instructions`: 执行逻辑与步骤
- `Output Format`: 输出格式规范

#### 步骤 2: 缺陷诊断 (Flaw Diagnosis)

**2.1 幻觉风险诊断**

检查目标 Skill 是否存在以下高风险模式：

| 风险等级 | 模式 | 描述 | 示例 |
|---------|------|------|------|
| 🔴 严重 | 无校验输出 | 直接输出结论无校验步骤 | "根据输入分析得出..." |
| 🔴 严重 | 假设性推断 | 无依据地进行推断 | "假设用户需要..." |
| 🟡 中等 | 模糊降级 | 未定义明确的降级策略 | "如果无法确定..." |
| 🟡 中等 | 缺失追问 | 未定义信息不足时的处理 | 无追问话术定义 |

**2.2 知识库脱节诊断**

检查目标 Skill 是否存在以下问题：

| 问题类型 | 描述 | 影响 |
|---------|------|------|
| 🔴 无检索指令 | 完全没有知识库检索步骤 | 依赖过时预训练数据 |
| 🔴 无关键词提取 | 未定义如何构建查询 | 检索效率低下 |
| 🟡 无来源标注 | 输出未标注信息来源 | 无法追溯验证 |
| 🟡 无未命中处理 | 未定义知识库无结果时的策略 | 可能返回无依据内容 |

#### 步骤 3: 信息补齐策略设计 (Information Gap Strategy)

**3.1 识别必需参数**

根据 Skill 的业务目标，确定必需的输入参数：

```markdown
## 必需参数清单

| 参数 | 用途 | 缺失影响 | 追问话术 |
|------|------|---------|---------|
| {param1} | {用途} | {影响} | "请问您能提供{param1}吗？这将帮助我..." |
| {param2} | {用途} | {影响} | "为了更准确地...，我需要了解{param2}" |
```

**3.2 设计追问话术框架**

```markdown
## 引导式追问框架

### 场景 1: {缺失场景1}
**触发条件**: 当用户输入缺少 {参数} 时
**追问话术**: "
  我注意到您没有提供 {参数} 信息。
  为了给您更准确的 {业务目标}，能否补充以下内容：
  - {具体问题1}
  - {具体问题2}
"

### 场景 2: {缺失场景2}
**触发条件**: 当 {条件} 时
**追问话术**: "
  基于{已有信息}，我需要进一步了解：
  - {具体问题}
"
```

#### 步骤 4: 知识库调用逻辑设计 (RAG Integration Plan)

**4.1 查询词提取策略**

```markdown
## 关键词提取规则

### 实体提取
- 公司/组织名: 提取完整名称
- 产品/模块名: 提取标准术语
- 时间范围: 标准化为日期格式

### 意图分类
- 分析类: 提取分析对象和维度
- 匹配类: 提取匹配条件和约束
- 生成类: 提取生成目标和格式要求

### 查询词构建
- 主查询: {实体} + {意图关键词}
- 扩展查询: {同义词} + {上下位词}
```

**4.2 知识库检索流程**

```markdown
## 知识库检索步骤

### Step 1: 检索准备
1. 解析用户问题，提取关键实体
2. 确定检索范围（案例库/产品文档/配置库）
3. 构建查询语句

### Step 2: 执行检索
使用以下工具进行检索：
- `mcp__plugin_claude-mem_mcp-search__search`: 搜索记忆库
- `Grep`: 搜索本地文档
- `Glob`: 定位相关文件

### Step 3: 结果验证
1. 检查结果相关性
2. 验证信息时效性
3. 多源交叉验证

### Step 4: 降级策略
当知识库未命中时：
1. 明确告知用户"知识库中未找到相关信息"
2. 提供"基于通用知识的参考"并标注
3. 引导用户提供更多上下文
```

## Best Practices

### 审计时机
- ✅ 新 Skill 开发完成后立即审计
- ✅ Skill 输出出现幻觉时及时审计
- ✅ 知识库大规模更新后重新审计
- ✅ 每月进行定期质量审计

### 审计深度
- **快速审计**: 仅检查 Instructions 中的高风险模式
- **标准审计**: 检查完整逻辑链和 RAG 集成
- **深度审计**: 包括输出样本分析和边界测试

### 优化优先级
1. 🔴 P0: 修复幻觉风险点
2. 🔴 P0: 添加知识库检索指令
3. 🟡 P1: 完善追问话术框架
4. 🟡 P1: 添加来源标注要求
5. 🟢 P2: 优化输出格式

## Notes

1. **只审计不修改**: 审计报告仅供参考，不自动修改原文件
2. **保持业务语义**: 优化时不能改变 Skill 的核心业务目标
3. **兼容性检查**: 确保 RAG 集成不破坏现有工作流
4. **人工复核**: 建议对 P0 级优化进行人工确认
5. **版本记录**: 保留优化前后的版本对比

# Jargon Decoder（技术黑话解码器）

## Purpose

作为资深解决方案导师，对销售和售前团队进行"战术指导"，帮助他们更好地理解和应对客户需求。

**核心功能**:
- 将客户的技术"黑话"翻译成销售能懂的"白话"
- **将非标准术语精准映射为优维标准产品功能**
- 为销售团队提供专业话术和商业机会分析
- 为售前团队提供技术拆解、架构建议和风险预警
- 识别显性需求背后的隐性技术债务
- 预判项目边界和技术陷阱

**核心价值**:
- ✅ **双向赋能**：同时辅导销售和售前两个团队
- ✅ **黑话解码**：把技术术语转化为生活化比喻
- ✅ **数据标准化**：为后续的自动匹配（requirement-matching）提供清洗后的标准术语输入
- ✅ **风险预警**：提前识别技术大坑和过度承诺风险
- ✅ **边界把控**：帮助团队识别需求陷阱

## When to Use

在以下情况下使用此技能：
- 销售从客户现场带回一段包含技术黑话的沟通记录
- 需要快速理解客户的技术需求和商业价值
- 售前需要评估技术可行性和实施难度
- 准备客户拜访前需要理解客户背景
- 需要识别需求中的隐性技术债务
- 评估项目风险和边界

## Capabilities

### 1. 术语标准化 (Term Standardization)

**核心能力**：将客户的非标准表达精准映射为优维标准产品功能，为后续自动匹配提供清洗后的数据输入。

**三种映射逻辑**：

#### 1.1 口语清洗 (Slang → Standard)
- **规则**：将非技术口语转化为运维标准术语
- **示例映射**：
  - `"看机器挂没挂"` → `Host_Availability_Monitor`（主机可用性监控）
  - `"服务器死了没"` → `Host_Status_Check`（主机状态检查）
  - `"网络通不通"` → `Network_Connectivity_Monitor`（网络连通性监控）
  - `"应用卡不卡"` → `Application_Performance_Monitor`（应用性能监控）
  - `"日志在哪看"` → `Log_Query_Platform`（日志查询平台）

#### 1.2 竞品/行业黑话 (Industry Jargon → Standard)
- **规则**：识别竞品特有词汇并映射为优维标准功能
- **示例映射**：
  - `"业务墙"` / `"蓝鲸大屏"` → `Business_Visualization_Dashboard`（业务可视化大屏）
  - `"作业平台"` → `Job_Execution_Platform`（作业执行平台）
  - `"配置平台"` → `CMDB_Configuration_Platform`（配置管理平台）
  - `"监控平台"` → `Integrated_Monitoring_Platform`（综合监控平台）
  - `"流程引擎"` → `ITSM_Workflow_Engine`（ITSM流程引擎）

#### 1.3 模糊需求 (Vague → Specific)
- **规则**：将宽泛描述具体化为明确的产品功能
- **示例映射**：
  - `"交易慢在哪里"` → `Transaction_Trace_Analysis`（交易链路追踪分析）
  - `"想看业务情况"` → `Business_Metric_Dashboard`（业务指标大屏）
  - `"自动化运维"` → `Automated_Operation_Platform`（自动化运维平台）
  - `"故障快速定位"` → `Fault_Root_Cause_Analysis`（故障根因分析）
  - `"资源管理"` → `Resource_Inventory_Management`（资源清单管理）

**映射输出格式**：
```json
{
  "original": "看机器挂没挂",
  "standard_feature": "Host_Availability_Monitor",
  "confidence": "High",
  "category": "Slang",
  "chinese_name": "主机可用性监控"
}
```

### 步骤 0: 术语标准化预处理 (Term Standardization)

**目标**：在分析客户记录之前，先提取并清洗所有技术关键词，输出结构化的术语映射表。

**0.1 提取技术关键词**

从 client_notes 中识别所有可能的技术术语、口语表达、竞品词汇：

```
扫描逻辑：
1. 识别技术黑话（K8s、LLM、CMDB、微服务等）
2. 识别口语表达（"看机器挂没挂"、"网络通不通"等）
3. 识别竞品词汇（"业务墙"、"蓝鲸"、"作业平台"等）
4. 识别模糊需求（"自动化运维"、"故障定位"等）
```

**0.2 执行术语映射**

对每个识别出的术语，执行以下映射逻辑：

```
映射规则：
1. 口语清洗 (Slang)：
   - 匹配模式：日常口语、非正式表达
   - 映射目标：标准运维术语
   - 置信度：根据上下文判断（High/Medium/Low）

2. 竞品/行业黑话 (Industry Jargon)：
   - 匹配模式：竞品特有词汇、行业通用黑话
   - 映射目标：优维标准产品功能
   - 置信度：根据词汇明确度判断

3. 模糊需求 (Vague)：
   - 匹配模式：宽泛描述、不明确需求
   - 映射目标：具体产品功能
   - 置信度：根据上下文推断（通常为 Medium）
```

**0.3 生成术语映射表**

输出结构化的映射表（Markdown 表格格式）：

```markdown
## 0. 术语标准化映射表 (System Output)

> **说明**：以下是从客户沟通记录中提取并标准化的术语映射，用于后续自动匹配。

| 原始黑话/口语 | 映射标准功能 | 中文名称 | 置信度 | 分类 | 备注 |
|:-------------|:------------|:--------|:------|:-----|:-----|
| 业务墙 | Business_Visualization_Dashboard | 业务可视化大屏 | High | Industry Jargon | 竞品术语 |
| 看机器挂没挂 | Host_Availability_Monitor | 主机可用性监控 | High | Slang | 口语表达 |
| 自动化运维 | Automated_Operation_Platform | 自动化运维平台 | Medium | Vague | 需进一步明确 |
```

**输出要求**：
- 每个术语必须包含：原始词、标准功能、中文名称、置信度、分类、备注
- 置信度分为：High（明确匹配）、Medium（推断匹配）、Low（不确定）
- 分类为：Slang（口语）、Industry Jargon（行业黑话）、Vague（模糊需求）

### 步骤 2: 生成销售辅导内容

#### 2.1 导师解读（用大白话概括）

**目标**：让销售快速理解客户在说什么，消除技术恐惧感。

**输出要点**：
- 用极简的生活化语言概括需求
- 避免使用新的技术术语
- 点明商业本质（省钱？赚钱？合规？效率？）

**示例风格**：
```
别被那些名词吓到了，其实客户就是想找个机器人帮他们回邮件。
```

---

#### 2.2 核心"黑话"速通卡

**目标**：挑选3-5个最关键的技术术语，用通俗比喻解释。

**输出格式**：
```markdown
* **[黑话术语]**：[生活化比喻/通俗解释]。
    * *导师提示：遇到这个词，你就把它想象成 [生活场景]，客户一听就懂。*
```

**比喻原则**：
- 使用生活中常见的场景（餐厅、工厂、交通、家居等）
- 避免用另一个技术术语解释技术术语
- 强调功能和价值，而非技术实现

**示例**：
```markdown
* **K8s（Kubernetes）**：就像是"集装箱码头的自动化吊车系统"，用来自动管理成百上千个软件服务器。
    * *导师提示：客户说K8s，你就理解为"自动化的服务器管家"就行。*
```

---

#### 2.3 杀手锏话术（Golden Sentences）

**目标**：提供1-2句"内行话"，让销售在客户面前显得专业。

**输出格式**：
```markdown
> *"下次见到客户，你试着这么说，能显得你很专业："*
> "[称呼]，您提到的 [黑话] 其实核心就是解决 [痛点]，我们在这个领域的经验能帮您实现 [价值]..."
```

**话术原则**：
- 自然引用客户提到的黑话
- 点明痛点和价值
- 展示专业度但不炫技
- 留下悬念，引导深入交流

---

#### 2.4 导师的下一步建议

**目标**：给销售一个明确的行动指引。

**建议类型**：
- 需要确认的关键信息（预算、决策链、时间节点）
- 需要挖掘的隐藏需求
- 需要规避的敏感话题
- 下一步推进动作（约技术交流、POC、方案讲解）

**输出格式**：
```markdown
* [具体的下一步行动建议]
* [需要向客户确认的关键问题]
```

### 步骤 3: 生成售前辅导内容

#### 3.1 需求透视（Technical Scope）

**目标**：帮助售前看清需求的全貌，包括客户没说但必须做的事。

**输出结构**：
```markdown
* **显性需求：** [提取明确的技术实体]
* **隐性债（关键）：** [指出客户没说但必须考虑的技术工作]
```

**显性需求**：
- 列出客户明确提到的技术模块
- 使用标准技术术语（不是生活化比喻）
- 标注技术类别（如LLM、K8s、CMDB等）

**隐性债识别原则**：
- 数据准备工作（数据清洗、标注、脱敏）
- 基础设施配套（网络打通、权限申请、环境准备）
- 集成开发工作（API对接、数据同步、认证集成）
- 运维保障工作（监控、日志、备份、容灾）

**示例**：
```markdown
* **显性需求：** LLM私有化部署、RAG知识库、企业微信集成
* **隐性债（关键）：** 虽然客户只想要个聊天框，但你得做一整套RAG数据管道（文档解析、向量化、知识库构建），这部分工作量占总工作量的60%。
```

---

#### 3.2 架构设计建议

**目标**：给出技术实现的可行路径。

**输出结构**：
```markdown
* **推荐路径：** [给出成熟的技术组合建议]
* **关键难点：** [指出技术整合最头疼的地方]
```

**推荐路径原则**：
- 优先选择成熟、稳定的技术方案
- 考虑客户的技术栈兼容性
- 平衡功能需求和实施成本
- 提供2-3个备选方案（高中低配）

**关键难点识别**：
- 技术选型难点（性能、兼容性、成本）
- 集成难点（老系统对接、数据打通）
- 实施难点（环境限制、权限申请、人员配合）
- 运维难点（监控、故障排查、升级）

---

#### 3.3 导师的红色预警（Risks）

**目标**：提前预警技术大坑，避免过度承诺。

**输出格式**：
```markdown
* ⚠️ **[风险类型]：** [具体的风险描述和应对建议]
```

**风险类型**：
- **过度承诺风险**：销售可能承诺的、但技术上做不到或成本太高的功能
- **技术可行性风险**：技术方案存在的未验证风险
- **实施风险**：项目实施过程中可能遇到的障碍
- **运维风险**：上线后可能出现的问题

**预警原则**：
- 明确告知什么"千万别承诺"
- 说明技术局限性（如AI准确率、系统性能极限）
- 提醒需要客户配合的事项
- 建议合理的期望值设定

**示例**：
```markdown
* ⚠️ **过度承诺风险：** 千万别答应"100%准确识别"。告诉销售，AI识别准确率在85%-90%是行业正常水平，需要人工复核机制。不要背生产事故的锅！
```

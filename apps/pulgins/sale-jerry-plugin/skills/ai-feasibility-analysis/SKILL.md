# Feasibility Analysis（AI需求落地可行性分析）

## Purpose

为售前团队提供标准化的AI需求可行性分析能力，基于Agent/Skill交付模式对客户AI应用需求进行系统性评估，生成可直接用于客户汇报的可行性分析报告。

**核心功能**:
- 自动读取Excel需求清单，提取需求信息并统计分析
- 参考样板文件格式，输出结构一致的可行性分析报告
- 基于Agent/Skill模式进行逐条需求可行性评估（三级分类）
- 设计角色驱动的Agent/Skill架构方案
- 制定分阶段落地计划（速赢→能力扩展→系统深度对接）
- 识别关键风险并给出对策建议

**核心价值**:
- **标准化**：统一的分析框架和输出格式，不同项目产出一致质量
- **高效性**：自动提取需求信息+AI分析+报告生成，从人工2-3天压缩至2-3小时
- **专业性**：基于Agent/Skill交付模式的专业技术评估视角
- **可落地**：输出含分阶段计划和资源估算的可执行方案

## When to Use

在以下情况下使用此技能：
- 客户提供了AI应用需求清单（Excel格式），需要评估落地可行性
- 售前阶段需要生成AI需求可行性分析报告供客户汇报
- 项目启动前需要系统评估AI需求的技术路径和交付计划
- 需要将客户需求映射到Agent/Skill交付模式的架构设计
- 需要识别哪些需求可快速落地、哪些需要深度定制

**触发关键词**：AI需求、AI场景、AI需求分析、AI应用需求、AI落地、可行性分析、需求评估、落地分析、Agent Skill交付、需求分类、交付计划、AI陪跑

## Capabilities

### 1. 需求数据自动提取

自动读取Excel需求文件，提取以下信息：
- 工作表列表和结构
- 每条需求的完整字段（序号、岗位、部门、角色、工作内容、AI需求、重要程度、难易程度等）
- 按岗位/重要程度/难易程度的多维统计
- 识别需求的字段结构差异（不同客户的Excel格式可能不同）

### 2. 三级可行性评估

基于Agent/Skill交付模式，对每条需求进行三级评估：

| 评估等级 | 标准 | 典型场景 |
|---------|------|---------|
| **可直接交付** | 平台现有能力覆盖，知识库+配置即可 | SOP问答、审批催办、知识检索、规则判断 |
| **Skill定制可交付** | 需开发定制Skill，不依赖外部系统API | 文档生成、合同审查、OCR识别、评分模型 |
| **需要深度定制** | 需MCP连接器+Skill+系统侧改造 | 多系统数据对接、审批流事件监听、深度分析 |

### 3. Agent/Skill架构设计

- 从角色出发设计Agent（每个Agent对应一个真实业务岗位）
- 为每个Agent规划专属Skill和共享Skill
- 设计Agent间的协作关系和数据流向
- 确定共享知识库和系统对接层

### 4. 分阶段落地计划

- 第一阶段（速赢）：知识库驱动+催办预警类，不依赖外部系统API
- 第二阶段（能力扩展）：文档生成+校验类，Skill定制开发
- 第三阶段（深度对接）：MCP连接器+多系统联动，深度定制需求

### 5. 风险识别与投入产出预估

- 识别关键技术风险和缓解措施
- 估算各阶段人周投入
- 量化预期产出和效率提升指标

## Parameters

| 参数 | 类型 | 必须 | 默认值 | 描述 |
|------|------|------|--------|------|
| requirement_file | string | ✅ | - | 需求Excel文件路径，支持.xlsx/.xls格式 |
| template_file | string | ✅ | - | 可行性分析样板文件路径（Markdown格式），用于对齐输出格式和评估标准 |
| customer_name | string | ❌ | 从文件路径或环境变量提取 | 客户名称，用于报告标题 |
| output_dir | string | ❌ | 需求文件所在目录 | 输出报告的保存目录 |
| platform_name | string | ❌ | Knodo/Javis | AI平台名称，用于报告中描述平台能力 |
| skip_stats | boolean | ❌ | false | 是否跳过统计汇总（仅逐条评估） |

## Instructions

### 执行角色与核心原则

**角色定位**：专业的AI解决方案架构师，擅长将客户业务需求映射到Agent/Skill技术交付模式。

**核心原则**：
- **格式对齐**：输出报告的结构、表格格式、评估标准必须与样板文件保持一致
- **逐条评估**：每条需求都必须有明确的评估等级和实现方式说明
- **角色驱动**：Agent设计从业务角色出发，而非从功能出发
- **务实可行**：分阶段计划必须考虑实际约束（API依赖、知识库准备等）

### 步骤 1：参数验证与文件读取

**1.1 验证输入文件**

```bash
# 验证需求文件存在且可读
test -f "{requirement_file}" && echo "OK" || echo "FILE_NOT_FOUND"
file "{requirement_file}"
```

**1.2 验证样板文件**

```bash
test -f "{template_file}" && echo "OK" || echo "FILE_NOT_FOUND"
```

**1.3 确定客户名称**

优先级：用户显式提供 > 从样板文件标题提取 > 从需求文件路径提取 > 环境变量 `JAVIS_WORKSPACE_NAME`

### 步骤 2：读取并分析需求文件

**2.1 读取Excel文件全部内容**

使用Python pandas读取Excel文件，提取所有工作表数据：

```python
import pandas as pd

excel_file = pd.ExcelFile(requirement_file)
for sheet_name in excel_file.sheet_names:
    df = pd.read_excel(excel_file, sheet_name=sheet_name, header=None)
    # 记录每个sheet的行数、列数、表头结构
```

**2.2 智能识别需求表结构**

需求Excel通常包含以下字段（字段名称可能有变体）：
- 序号/编号/ID
- 岗位/角色/重点岗位
- 部门/所属部门
- 工作内容/业务场景
- AI需求/需求描述
- 重要程度/优先级
- 难易程度/复杂度

自动识别策略：扫描第一行或前几行，匹配常见字段名称变体，建立字段映射。

**2.3 统计分析**

自动生成以下统计：
- 总需求数
- 按岗位/部门的分布（数量、占比）
- 按重要程度的分布
- 按难易程度的分布
- 交叉分析（岗位 × 重要程度、岗位 × 难易程度）

### 步骤 3：读取并解析样板文件

**3.1 解析样板结构**

分析样板文件的章节结构，提取以下关键信息：
- 章节标题层级和命名规范
- 评估标准的等级名称和定义
- 逐条评估表的列结构
- Agent架构设计部分的组织方式
- 分阶段计划的时间粒度和命名

**3.2 建立映射规则**

根据样板中的评估标准，建立映射规则：

| 样板评估等级 | 映射到本项目的标准 |
|------------|-----------------|
| 样板中的一级评估名称 | 直接复用（如"可直接交付""Skill定制可交付""需要深度定制"） |
| 样板中的Knodo实现方式 | 替换为当前项目使用的平台名称 |
| 样板中的Javis参考Skill | 根据当前项目可用的Skill列表替换 |

**3.3 提取样板的Agent设计模式**

分析样板中的Agent设计模式（如角色命名、Skill命名规范、协作关系图格式），确保输出报告的设计风格一致。

### 步骤 4：逐条可行性评估

**4.1 评估每条需求**

对需求表中的每条需求，按以下逻辑评估：

```
IF 需求仅需知识库+RAG+规则引擎（无外部系统依赖）:
    评估等级 = "可直接交付"
    实现方式 = "工作空间知识库 + RAG检索 + 规则引擎 + 消息推送"
    前提条件 = "知识库素材导入"

ELIF 需求需要定制Skill但不需要外部系统API:
    评估等级 = "Skill定制可交付"
    实现方式 = "定制 `{skill-name}` Skill：{核心能力描述}"
    前提条件 = "相关模板/规则/知识库准备"

ELIF 需要外部系统API对接（读取/写入/事件监听）:
    评估等级 = "需要深度定制"
    实现方式 = "Skill逻辑 + MCP {系统名} 连接器 + Agent {自动任务描述}"
    前提条件 = "{系统名} API开放"

ELIF 需要专有模型训练或复杂算法:
    评估等级 = "需要深度定制"
    实现方式 = "定制 {skill-name} Skill：{核心能力} + {模型/算法描述}"
    前提条件 = "训练数据积累 + 模型训练环境"
```

**4.2 Skill命名规范**

Skill命名采用 kebab-case 格式，语义化命名：
- 文档生成类：`{target}-doc-gen`（如 `tender-doc-gen`）
- 审查校验类：`{target}-check` 或 `{target}-review`（如 `invoice-check`）
- 问答检索类：`{domain}-qa` 或 `{domain}-knowledge`（如 `legal-qa`）
- Agent自动任务类：`{action}-{engine}`（如 `approval-escalation`）
- 分析预测类：`{domain}-analysis` 或 `{domain}-predict`（如 `bid-score-predict`）

**4.3 按岗位分组输出**

将逐条评估结果按岗位/角色分组，每组使用独立的评估表格，表格列与样板保持一致：
| 序号 | 工作内容 | AI需求 | 评估 | {平台}实现方式 | 前提条件 |

### 步骤 5：Agent/Skill架构设计

**5.1 从岗位出发设计Agent**

- 每个Agent对应需求表中的一个"重点岗位/角色"
- Agent命名：`{岗位名称}Agent`（如"行政主管Agent"）
- Agent定位描述：一句话概括该Agent的职责范围

**5.2 为每个Agent规划Skill**

按以下分类组织每个Agent的Skills：

| 类别 | 说明 | 示例 |
|------|------|------|
| 专属Skill | 仅该Agent使用的Skill | `tender-doc-gen`仅售前Agent |
| 共享Skill | 多个Agent复用的Skill | `sop-qa`被多个Agent共享 |
| Agent自动任务 | 需要定时/事件触发的自动化任务 | 审批催办、节点预警 |

**5.3 设计Agent协作关系**

绘制ASCII协作关系图，展示：
- Agent之间的数据流向
- 共享知识库
- 系统对接层（MCP连接器）
- 项目生命周期各阶段的Agent职责

### 步骤 6：统计汇总

**6.1 按可行性分类统计**

| 可行性等级 | 需求条数 | 占比 | 对应序号 |
|-----------|---------|------|---------|

**6.2 按岗位分类统计**

| 岗位 | 总数 | 可直接交付 | Skill定制 | 深度定制 | 落地难度 |
|------|------|----------|----------|---------|---------|

**6.3 交叉分析**

| 可行性 × 重要程度 | 高 | 中 | 低 | 合计 |
|-------------------|---|---|---|------|

### 步骤 7：分阶段落地计划

**7.1 第一阶段：速赢（建议2-4周）**

交付策略：知识库驱动 + 催办预警 + 规则判断类需求
入选标准：评估等级为"可直接交付"的全部需求

**7.2 第二阶段：能力扩展（建议4-8周）**

交付策略：Skill定制开发，文档生成/校验/分析类需求
入选标准：评估等级为"Skill定制可交付"的全部需求

**7.3 第三阶段：深度对接（建议8-16周）**

交付策略：MCP连接器开发 + 系统深度集成
入选标准：评估等级为"需要深度定制"的全部需求
关键依赖：外部系统API开放

### 步骤 8：风险与投入产出

**8.1 风险识别**

按以下维度识别风险：
- 系统API依赖风险
- 知识库质量风险
- 多系统数据打通风险
- 用户信任度风险
- 项目范围管理风险

**8.2 投入预估**

| 阶段 | 我方投入 | 客户投入 |
|------|---------|---------|

单位：人周。估算依据：
- 可直接交付的Skill：0.5人周/个
- Skill定制：1-2人周/个
- 深度定制：2-4人周/个
- MCP连接器开发：2-3人周/个
- Agent骨架搭建：0.5人周/个

**8.3 产出预估**

| 指标 | 当前状态 | 第一阶段后 | 第二阶段后 | 第三阶段后 |
|------|---------|----------|----------|----------|

## Notes

1. **样板优先**：当提供了样板文件时，输出格式必须严格对齐样板。样板中的评估标准、章节结构、表格列名是最高优先级的格式参考
2. **逐条无遗漏**：需求表中的每一条需求都必须有评估结论，不允许批量跳过或概括性描述
3. **Agent数量由数据决定**：Agent数量 = 需求表中"岗位/角色"字段的去重值数量，不要预设固定的Agent数量
4. **Skill命名一致性**：同一Skill在不同Agent中引用时必须使用相同名称，共享Skill需在汇总中标注
5. **分阶段计划务实性**：第三阶段的周期必须考虑系统API对接的实际难度，不要低估
6. **前提条件明确**：每条"需要深度定制"的需求都必须明确写出需要哪个系统开放什么API
7. **样板中的平台名称替换**：样板中出现的"Knodo""Javis"等平台名称需替换为当前项目实际使用的平台名称

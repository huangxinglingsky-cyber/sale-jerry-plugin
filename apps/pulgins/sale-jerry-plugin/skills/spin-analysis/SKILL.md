# SPIN Analysis (SPIN销售法分析技能)

## Purpose

基于SPIN销售方法论（Situation-Problem-Implication-Need-payoff）分析会议纪要或客户对话，识别客户现状、痛点、影响和需求，判断销售阶段。

**核心功能**:
- S维度分析（Situation 现状信息）
- P维度分析（Problem 痛点问题）
- I维度分析（Implication 影响后果）
- N维度分析（Need-payoff 需求期望）
- 销售阶段智能判断（8个阶段）
- 推进建议生成

**核心价值**:
- ✅ **方法标准化**：基于经典SPIN销售法
- ✅ **真实性第一**：严格基于会议内容，禁止虚构
- ✅ **阶段精准**：准确判断当前销售阶段
- ✅ **建议可行**：提供具体可操作的下一步行动
- ✅ **全局视角**：综合分析客户采购心理

## When to Use

- 分析会议纪要理解客户需求
- 判断当前销售阶段
- 制定客户跟进策略
- 识别客户痛点和期望
- 生成销售推进建议

## Capabilities

### 1. S维度分析（现状信息识别）

**识别能力**:
- 提取客户当前使用的系统和工具
- 识别客户的技术架构和技术栈
- 分析团队规模和组织结构
- 评估业务规模和运维压力
- 理解现有流程和规范

**关键输出**:
- 客户现状信息清单
- 系统/工具清单
- 组织结构描述

### 2. P维度分析（痛点问题识别）

**识别能力**:
- 提取明确表达的问题和困难
- 识别效率低下的环节
- 发现频繁发生的故障
- 分析人力成本压力
- 评估工具不足或缺陷

**关键输出**:
- 痛点问题清单
- 问题严重程度评估
- 影响范围描述

### 3. I维度分析（影响后果分析）

**识别能力**:
- 分析问题造成的业务影响
- 量化成本损失（时间、人力、金钱）
- 识别隐藏的风险隐患
- 评估对KPI的影响
- 理解长期后果

**关键输出**:
- 业务影响分析
- 成本损失量化
- 风险评估

### 4. N维度分析（需求期望明确）

**识别能力**:
- 提取明确提出的需求
- 理解期望的改善目标
- 识别对解决方案的期待
- 评估预期的价值收益
- 判断优先级

**关键输出**:
- 需求清单
- 期望目标描述
- 价值期待

### 5. 销售阶段智能判断

**判断能力**:
- 基于SPIN四维度信息判断当前阶段
- 识别8个销售阶段特征
- 评估阶段推进的充分性
- 计算阶段判断置信度

**支持的8个阶段**:
1. 初步接触 → 2. 需求探索 → 3. 需求明确 → 4. 方案呈现
5. 方案评估 → 6. 商务谈判 → 7. 决策推进 → 8. 签约成交

### 6. 推进建议生成

**建议能力**:
- 基于当前阶段生成具体下一步行动
- 提供关键里程碑提示
- 识别遗漏的关键信息
- 评估推进风险和机会
- 优化销售策略

## Instructions

### 执行角色与核心原则

**你的角色定位**：
你是一位精通SPIN销售方法法的销售顾问，擅长从会议纪要中提取客户需求信息。

**你的任务目标**：
基于SPIN方法论分析会议纪要，识别S-P-I-N四维度信息，判断销售阶段。

**核心原则**：
1. **追问优于编造**：信息不足时必须标注，禁止虚构SPIN信息
2. **检索优于记忆**：优先检索知识库获取SPIN定义，避免依赖预训练
3. **来源必须标注**：所有SPIN信息必须标注原文来源

### 步骤 0: 输入完整性校验（必须执行）

**⚠️ 重要：在进行SPIN分析前，必须先校验输入有效性**

**0.1 检查输入是否有效**

| 检查项 | 条件 | 处理方式 |
|-------|------|---------|
| 非空检查 | `conversation_text` 不能为空 | 返回错误，要求提供会议内容 |
| 长度检查 | 长度 ≥ 50 字符 | 不足则追问用户提供更完整内容 |
| 相关性检查 | 包含 ≥ 3 个SPIN关键词 | 不足则警告分析可能不完整 |

**0.2 SPIN关键词检测**

检测输入是否包含以下关键词（至少3个）：

| 维度 | 关键词示例 |
|------|-----------|
| S (Situation) | 目前、现在、当前、使用、正在、现有 |
| P (Problem) | 问题、困难、痛点、不足、缺陷、经常、频繁 |
| I (Implication) | 影响、导致、造成、损失、风险、后果 |
| N (Need-payoff) | 需要、希望、期望、想要、计划、目标 |

**0.3 校验结果处理**

```json
// 校验通过
{
  "status": "validated",

  "spin_keywords": {"S": 2, "P": 3, "I": 1, "N": 2},

// 校验警告 - 信息可能不足
{
  "status": "warning",
  "text_length": 80,

  "message": "输入文本较短且SPIN关键词不足，分析结果可能不完整",
  "suggestion": "建议提供更完整的会议纪要（至少50字，包含现状、问题、需求等描述）"

// 校验失败 - 输入无效
{

  "message": "会议纪要内容过短，无法进行有效的SPIN分析",
  "suggestion": "请提供至少50字的会议纪要或客户对话内容"
```

### 步骤 0.5: 知识库检索（SPIN方法论）

**⚠️ 重要：在进行SPIN分析前，优先检索知识库获取标准定义**

**0.5.1 执行检索**

```bash
# 检索SPIN方法论定义
# 方式1：搜索本地知识库
Grep(pattern="SPIN|Situation|Problem|Implication|Need", path="knowledge/")

# 方式2：搜索产品相关目录
Glob(pattern="**/*SPIN*.md")
```

**0.5.2 SPIN标准定义参考**

如果知识库检索成功，使用检索到的定义作为分析基准：

| 维度 | 英文 | 中文定义 | 识别要点 |
|------|------|---------|---------|
| S | Situation | 客户当前状况和背景 | 现有系统、团队、流程、规模 |
| P | Problem | 客户面临的问题和困难 | 痛点、困难、不足、缺陷 |
| I | Implication | 问题不解决将导致的后果 | 影响、损失、风险、后果 |
| N | Need-payoff | 解决问题后能获得的价值 | 需求、期望、目标、收益 |

**0.5.3 知识库未命中降级**

如果知识库未找到相关定义，使用通用定义：
> [标准SPIN定义] 并在输出中标注"基于通用知识，建议结合知识库验证"

**0.5.4 标注信息来源**

在输出报告中必须标注：
```markdown
**SPIN方法论来源**: {知识库路径 | 基于通用知识}
```

**识别内容**:
- 明确表达的问题和困难
- 效率低下的环节
- 频繁发生的故障
- 人力成本压力
- 工具不足或缺陷

**识别内容**:
- 问题造成的业务影响

- 风险隐患
- 对KPI的影响

**关键词**:

**输出格式**:

```markdown
- 运维团队加班严重，人员流失风险高
```

### 步骤 4: N维度分析（需求期望）

**识别内容**:
- 明确提出的需求
- 期望的改善目标
- 对解决方案的期待
- 预期的价值收益

**关键词**:
"需要"、"希望"、"期望"、"想要"、"计划"

**输出格式**:
```markdown

- 希望实现自动化配置发现
- 期望故障恢复时间缩短到30分钟以内
- 计划减少50%的人工运维工作量
```

---

### 步骤 5: 销售阶段判断

**8个销售阶段**:

| 阶段 | 特征 | SPIN表现 |
|------|------|---------|
| 1. 初步接触 | 建立联系 | S信息较多，P/I/N较少 |
| 2. 需求探索 | 挖掘痛点 | P信息增多，I开始出现 |
| 3. 需求明确 | 痛点清晰 | P/I信息充分，N开始明确 |
| 4. 方案呈现 | 展示方案 | N信息完整，讨论解决方案 |
| 5. 方案评估 | 客户评估 | 讨论方案细节、可行性 |
| 6. 商务谈判 | 价格讨论 | 讨论预算、合同、时间 |
| 7. 决策推进 | 促成签约 | 讨论采购流程、决策人 |
| 8. 签约成交 | 合同签订 | 确认合同细节 |

**判断逻辑**:
```
如果 S信息充分 且 P信息较少:
  阶段 = "初步接触"

如果 P信息充分 且 I信息充分 且 N信息明确:
  阶段 = "需求明确"

如果 讨论解决方案细节:
  阶段 = "方案呈现" 或 "方案评估"

如果 讨论价格和合同:
  阶段 = "商务谈判" 或 "决策推进"
```

### 步骤 6: 推进建议生成

**基于当前阶段给出下一步行动**:

```
初步接触 → 深入挖掘痛点，进入需求探索
需求探索 → 量化影响，明确需求
需求明确 → 准备方案，安排方案讲解会议
方案呈现 → 收集反馈，准备POC
方案评估 → 推动商务谈判
商务谈判 → 推动决策流程
决策推进 → 跟进合同签署
```

**输出格式**:
```markdown
## 📈 销售推进建议

### 当前阶段
**需求明确阶段** - 客户痛点清晰，需求明确

### 下一步行动
1. 准备针对性的解决方案PPT
2. 安排方案讲解会议，邀请运维团队参加
3. 准备同行业成功案例（XX银行、XX证券）
4. 预估ROI数据，准备量化价值呈现

### 关键里程碑
- ✅ 已完成：需求调研
- 🔄 进行中：方案准备
- ⏳ 待推进：方案讲解会议
```

## Best Practices

### 使用最佳实践

1. **真实性第一**:
   - 所有SPIN信息必须来自会议纪要原文
   - 严禁虚构客户痛点或需求
   - 如信息不足，标注"信息缺失"

2. **客观判断**:
   - 基于充分证据判断销售阶段
   - 标注判断置信度
   - 避免主观臆测

3. **建议可行**:
   - 下一步行动具体明确
   - 考虑实际可行性
   - 提供多个选项
